> ### 왜 GPT로 만들게 되었을까?---나는 개발자지만 **AI Agent에 대한 부정적인 시선**은 갖고 있지 않다.
> (개발자들이 부정적이라는 뜻이 아니라, 그럴 수도 있다는 의미 ❌)오히려 나는 GPT를 **정말 유용한 ‘설명 도우미’**라고 생각한다.
> 설명을 잘 못하는 편인데, ChatGPT에게 “누군가에게 설명할 수 있게 적어줘”라고 말하면
> **발표 대본을 뚝딱 써주는 선생님** 같은 존재가 된다.하지만 현재 내가 쓸 수 있는 AI는 **ChatGPT밖에 없다.**
> 그 이유는 단순하다:- 가장 먼저 접한 AI였고- 사이트만 들어가면 쉽게 쓸 수 있는 **무료 웹 AI**이기 때문이다.그리고 이 블로그는 사실 **ChatGPT가 아니라 Claude Desktop**으로 만들었다.
> 한마디로, “세상이 너무 좋아졌다”는 감상만 남는다.그런데 왜 굳이 Markdown Editor를 만들지 않고 GPT를 사용했냐고 묻는다면…
> **Claude Desktop은 이미 잠들어 있었다.** 새벽 6시까지 😢그래서 이번 기회에 **ChatGPT로 구현해본다면**
> 블로그를 만들 때와 어떤 차이가 있는지 체감할 수 있을 것 같았다.
> 그래서 가장 단순한 기능인 **Markdown Editor**를 GPT로 구현해보기로 했다.

## 1️⃣ 첫 번째 시도: 최소한의 구조로 빠르게 만들기

처음에는 **GitHub Pages에 올릴 수 있는 가장 단순한 형태**로 접근했다.
목표는 딱 하나였다.

> **“최소한의 구조로, 최대한 빠르게 동작하는 Markdown Editor”**

그래서 1단계 MVP는 다음 조건을 만족하도록 설계했다.

```shell
🎯 목표 (1단계 MVP)
- 좌: 마크다운 입력
- 우: 실시간 미리보기
- 자동 저장 (localStorage)
- 새로고침해도 내용 안 날아감
- 빌드 없음 / 프레임워크 최소
```

구조도 최대한 단순했다.

```shell
🧱 구조 (이대로 만들면 됨)
/playground/markdown-editor/
  index.html
  app.js
  style.css
```

> 💡 포인트는 **“프레임워크 없이도 동작하는 최소 MVP”**였다.

---

### 첫 번째 시도 Retro: “React 블로그와의 충돌”

만약 내가 Jekyll 같은 정적 사이트 생성기로 배포했다면,
이 코드만 그대로 복사해서 붙여넣으면 끝이었다.

하지만 나는 **React + Vite 기반 블로그**를 운영하고 있었고,
그 구조와는 코드가 맞지 않았다.

이때 Claude와 ChatGPT의 차이를 체감했다.

- Claude는 **내 로컬 코드에 접근 가능**해서
프로젝트 구조에 맞게 자동으로 수정/생성할 수 있었을 것이다.
- ChatGPT는 구현은 해주지만
내 프로젝트 구조를 자동으로 맞춰주지는 않는다.

그래서 결국 나는 **React 기반으로 다시 만들기로 결심**했다.

---

## 2️⃣ 두 번째 시도: React 안에서 다시 설계하기

React로 재구성하기로 마음먹고,
이번에는 전체 구조를 이렇게 정의했다.

> **블로그 = Shell / Playground = 내부 라우트 앱들**

예를 들면 다음과 같은 구조다:

- `/` (블로그 글)
- `/playground`
- `/playground/markdown`
- `/playground/2048`
- `/playground/tictactoe`

즉, **React 앱 안에 미니 앱들을 넣는 형태**다.

---
GPT의 답변은 다음과 같았다.

### 🎯 React Markdown Editor MVP

React로 구현할 때 필요한 핵심 스택은 다음과 같다.

- `react-markdown` (미리보기)
- `remark-gfm` (표, 체크박스)
- `localStorage`
- 상태 = React state

패키지 설치는 이렇게 진행했다.

```shell
npm install react-markdown remark-gfm
```

---

### 두 번째 시도 Retro: “코드는 잘 나오지만, UI는 내 블로그랑 다르다”

ChatGPT는 React에 맞춘 코드를 제공했고,
그 과정을 따라한 결과 **정말 완성된 Markdown 페이지**를 만들 수 있었다.

하지만 문제는 다음이었다.

- Router 설정이 내 블로그 구조와 맞지 않았다.
- 테마가 내 블로그 스타일과 다르게 느껴졌다.
- 즉, 기능은 완벽했지만 **UI가 내 블로그와 결이 달랐다.**

그럼에도 불구하고, 중요한 사실은 변하지 않는다.

> **“코드를 한 줄씩 작성하지 않아도 기능 구현이 가능하다.”**

## 마무리 (소감) ✨

이번 경험에서 얻은 결론은 단순하다.

> GPT를 사용하면 **아이디어를 구현하는 속도**가 확실히 빨라진다.

다만, **내 블로그의 구조와 디자인에 완벽히 맞추려면**
AI가 만든 코드에서 한 번 더 손봐야 한다는 점도 확인했다.

그래도 이 과정은 충분히 가치 있었다.
